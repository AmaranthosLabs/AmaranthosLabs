Build from nothing

2022-09-01

Where do I start?

We must be able to start from nothing. This means no external dependencies. We want simplicity of scale, therefore we need to abstract. We also want something that can begin as a commodity and develop into more. We need to define the way WE see the universe.

In OOP, you may start thinking of Objects.

In DDD you may start looking for Aggregates.

In Event Sourcing you may start looking at Events.

For a data center, we have 3 particular areas of convergence:

  1. Network
  2. Compute
  3. Storage

These 3 areas form the fundamental idea of a center for information access and data center infrastructure management, they form the Bounded Contexts of infrastructure. While we may have been taught to think of these ideas as classes, to the composable information machine they are simply boundaries. For us, their definition is also an axiom, something we defined and accept without further proof. These end up as Things as defined below and also lead us later into a proof system.

In a Composable Information Machine, we have 3 base abstractions:

  1. Person
  2. Place
  3. Thing

NOTE: this is different than inheritance. In Rust, we use Traits and Implementations to define constraints, behaviors and relationships.

These are not an encapsulation as in Object Oriented Classes. We are defining the 3 basic ideas that will permeate the entire system we build. We don't derive everything from Object as in Object Oriented Programing.

We want to clarify where boundaries lie, not necessarily shove everything into a hierarchy or encapsulation. We will have to get used to this idea if coming from a purely OOP background. Functional Programming is a bit different.

We will define some more Axioms, meaning a definition, so we understand exactly what we are working with. In DDD you might calls these Aggregate Roots.

Person

A person is human being.

People are the plural of person. It can be a lot of other things, but it roots as a human being. Properties are not widely defined in this structure, relying on traits and implementations to build utility.

We will be avoiding definitions such as this, though we can project into them and read from them through the use of ACLs.

We support things acting as humans, but they are not people, we will talk about this more in identification, authentication, and authorization.

Place

A place in a defined location in the universe.

Anything added to that is a metric, or topology, or something else which are traits.

Thing

A definable concept.

These are the definitions we will use to begin, everything is evolutionary from here.

We have defined these in Rust.
